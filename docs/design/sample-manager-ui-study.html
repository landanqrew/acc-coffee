Research summary

What gives sample-manager its look

sample-manager has a fully documented, locked design language (its docs/design/design-philosophy.md + a reference mockup). Below is everything I extracted — the philosophy, the exact tokens, the component grammar, and the navigation rules — rendered in that language so you can see it, not just read it. Each section ends with a note on what it means for acc-coffee, which is mobile-first where sample-manager is desktop-first. We'll turn this into a concrete re-skin plan.

Stack match is in our favor. Both apps are Next.js + React 19 + Tailwind v4. sample-manager adds shadcn/ui (new-york, neutral base) + lucide icons + class-variance-authority/tailwind-merge. acc-coffee currently ships the default Geist/neutral theme and no component library — so adopting this language is mostly a token re-theme + a small set of shared primitives, not a rewrite.

The design DNA

Five principles do all the work. They're worth internalizing because they're rules, not vibes — each one tells you what to do at a decision point.

1 · Separate shell from data

The friendly-vs-dense tension is resolved by zoning. The shell (headers, cards, nav) is roomy, soft, accented. The data core (tables, grids) is tight, hairline, flat. Keep the seam between them clean.

2 · Depth, not borders, separates the shell

White cards float on a cool-gray app background via lift shadows — they don't sit in outlined boxes. Borders are reserved for hairline dividers inside dense regions.

3 · Color carries meaning, not decoration

Royal blue = interactive (links, active nav, primary buttons). Status green/amber/red = state only. No decorative color beyond the accent — neutral surfaces keep status legible.

4 · Type split is the type system

Sans (DM Sans) for language; mono (JetBrains Mono) for data — IDs, prices, codes. Numbers and identifiers always go mono so columns align and scan.

5 · Edit-depth picks the surface

A record's own properties (direct object) → side-pane overlay. Drilling into a child record → full page. The surface is chosen by relationship depth, not by how big the record is.

Navigation & editing model

This is the part most worth a conscious decision for acc-coffee, because it's the most desktop-biased.

Hover-expand rail

Left nav is a 64px icon rail that expands to 240px labels on hover/focus — as an overlay, with no content reflow. Active item gets a 3px cyan bar + accent tint.

Breadcrumb + ⌘K top bar

A 56px top bar carries breadcrumbs (tracking edit depth), a global ⌘K search affordance, and a QBO sync status chip.

Side-pane for direct objects

Editing a row's own props (or "New X") opens a right side-pane overlay (≤440px, panel shadow, dimmed backdrop) — the list stays in context.

Full page for child records · infinite scroll

Drilling into a child (job → samples) is full-page navigation. Lists use infinite scroll, never pagination, with a sentinel row.

Mobile responsiveness — the honest assessment

Key finding: sample-manager is built desktop-first. Its components carry almost no responsive breakpoints — fixed-width rail (w-16→w-60), a 320px search box, a w-80 drawer, wide multi-column tables, and a h-svh non-scrolling shell. It looks the way it does at desktop width. So we inherit its visual language, but we must design the responsive behavior ourselves — acc-coffee is mobile-first.

The good news: the aesthetic (tokens, type split, pills, lifted cards, status colors) is fully portable to mobile. Only the navigation shell and dense tables need a mobile rethink. Here's the proposed translation, rendered in a phone frame:

Mobile-first translations (proposed)
  • Hover-expand rail → bottom tab bar (thumb reach) on mobile; the rail returns at md:+.
  • Record lists → card list everywhere; wider cards / multi-column grid at md:+ (no separate desktop table — acc-coffee's data isn't dense).
  • Right side-pane → bottom sheet with a drag handle on mobile; right pane at md:+.
  • ⌘K bar → an icon button opening a full-screen search on mobile.
Mobile-first hygiene to add
  • Tap targets ≥ 44px; bump input padding to py-2.5/3 and font to text-base (prevents iOS zoom).
  • Respect safe-area insets (env(safe-area-inset-bottom)) on the tab bar / sheet.
  • Tighten the desktop densities a touch — the data core is very tight for thumbs.
  • Keep the lifted-card aesthetic — it reads beautifully on mobile already.

Implementation roadmap (finalized)

The full "Elevated" grammar is the target, delivered in phases. Foundation lands first, then one tracer-bullet slice proves the whole stack, then the remaining screens go in parallel. Each item below is tracked as its own GitHub issue.

Phase 0 · Foundation (sequential)
  1. 1Theme tokens + fonts + shadcn init. Write the royal-blue token contract into globals.css (mapped to shadcn semantic tokens); DM Sans / JetBrains Mono via next/font; install shadcn/ui + cva + tailwind-merge + lucide-react + a cn() helper. Light-only. No screens change yet.
  2. 2Core primitives — Button, Card, Pill, StatusPill, Field — themed to the tokens.
Phase 1 · Shell + tracer-bullet slice
  1. 3App shell — mobile bottom tab bar (Services · Stock · Runbook · More) + More sheet + desktop rail, role-aware. Retire /dashboard; Services becomes the home, absorbing the "Today's coffee" view.
  2. 4Re-skin Stock + Stock-Count bottom sheet — the reference slice: lifted cards, mono counts, Low/Out StatusPill, and the sheet edit pattern end-to-end.
Phase 2 · Remaining screens (parallel after 1–4)
  1. 5Services list — Today / Upcoming / Past card groups; Lead schedule & special-event forms.
  2. 6Service detail + Service Report form — full page; brew quantities inline; questions + Stock Counts; feedback summary.
  3. 7Supplies + Supply add/edit sheet (Lead).
  4. 8Runbook — sections, markdown, full-page section editor.
  5. 9Feedback results — average-rating stat cards + comments.
  6. 10Settings + Team (Lead).
  7. 11Public / auth — sign-in, check-email, and the congregant Feedback Survey.
Finalized for acc-coffee (resolved in design review):
  • Accent: church royal blue — #2f6bff primary / #1d4ed8 deep for AA text & links.
  • Primitives: shadcn/ui + Radix + cva + tailwind-merge + lucide-react, adopted up front (one styling idiom).
  • Mobile nav: bottom tab bar — Services (home) · Stock · Runbook · More; Dashboard retired and folded into Services; Feedback lives in More; desktop rail shows the full list.
  • Theme: light-only.
  • Edit model: bottom sheets for quick single-object edits (Stock Count, Supply add/edit); full pages for the Service Report & Runbook; brew quantities stay inline on the Service.
  • Type: DM Sans for language, JetBrains Mono for numbers / times / IDs only.
  • Data layout: responsive card lists everywhere — no separate desktop tables (our data isn't dense).
